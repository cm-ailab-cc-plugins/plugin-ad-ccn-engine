# Ad CCN Engine — 通用素材切角引擎

這個 Skill 幫使用者（多為無技術背景的行銷／產品人）系統化地發想、測試、追蹤
**任何 App／產品**的投放廣告素材。核心方法是 CCN 框架（見 `ccn-framework.md`）。

**這是一個「方法引擎」，本身不綁任何產品。** 你把產品資訊貼進來，它就幫你長出
那支產品專屬的 TA、切角、策略與記憶。

---

## 這個 Skill 解決什麼問題

1. **一套引擎、無限個產品**：方法（CCN、流程、命名、停損）寫在引擎；產品資料各自獨立。
2. **每次不用重新跟 AI 解釋產品**：每支產品的 product-profile.md 存好資訊、賣點、4 個 TA。
3. **AI 有記憶、不會重複生切角**：每支產品的 matrix-state.md 記錄生過的切角和測試結果。
4. **有邏輯地生素材**：用 CCN 框架把一個切角翻譯成四種 TA 講法，不是亂槍打鳥。
5. **可視覺化**：每次更新都輸出「可貼上 Google Sheet」的表格。

## 檔案架構（很重要，先搞懂）

### 引擎本體（方法，幾乎不動）
```
ad-ccn-engine/
  ├─ SKILL.md              主流程＋指令對應（本檔）
  ├─ ccn-framework.md      CCN 方法論 ＋ 通用切角原型庫
  └─ templates/
        ├─ product-profile.template.md   產品檔空白範本
        └─ matrix-state.template.md      記憶檔空白範本
```

### 每支產品的工作區（資料＋記憶，放桌面）
```
桌面/切角引擎/<產品名>/
  ├─ product-profile.md    這支產品：資訊＋賣點＋4 個 TA＋切角庫＋投廣策略
  └─ matrix-state.md       這支產品的記憶：靈感庫 + 測試結果矩陣
```

> **工作區根目錄固定為** `桌面/切角引擎/`。每支產品一個子資料夾，互不干擾。
> 動作前一定要先確認「現在在哪支產品」，避免把 A 產品的資料寫到 B 產品。

## 核心運作原則（每次執行都要遵守）

### 原則 1：先確認在哪支產品
任何動作前，先確認目前操作的是哪支產品（哪個工作區資料夾）。
若不清楚，先問使用者，或列出 `桌面/切角引擎/` 底下有哪些產品讓他選。

### 原則 2：動作前先讀記憶
任何「發想」動作，**第一步一定先讀該產品的 `matrix-state.md`**，確認：
- 哪些切角已經生過（不重複生）
- 哪些正在測試中
- 哪些已經淘汰

### 原則 3：不自動寫檔，等使用者指令
發想完切角、或整理完產品資料，**都不會自動存**。要等使用者明確說「存」才寫檔。
（避免每次的草稿都被記進去，造成混亂。）

### 原則 4：每次寫檔後，輸出可貼上 Google Sheet 的表格
更新任何 .md 之後，另外輸出一段乾淨的表格，讓使用者直接複製貼進 Google Sheet，
並明確標示要貼到哪個分頁。

### 原則 5：白話、不丟術語
使用者多半無技術背景。每次動作前先一句話說「我要做什麼」，做完講「結果是什麼、
下一步建議」。不丟 Meta 內部術語。

### 原則 6：寫外部資料要「先預覽、只動該動的、寫完驗證」
牽涉寫 Google Sheet 或動廣告帳戶時：**先給使用者預覽、經同意才寫**；只更新該更新
的欄位、其他一律不碰；**寫前備份、寫後讀回驗證**；**認廣告 ID 不認名字**（Sheet 命名
與 matrix 命名常對不上，只有 ID 靠得住）。動錢有關的（關廣告、改預算）**只建議、
不自動執行**，留一道人工關卡。

## 指令對應表

### 🆕「新增產品」（＋使用者貼上產品資訊）——一條龍上線流程

這是把「任何產品」接進引擎的入口。使用者貼的資料可以很亂（App 連結、一段介紹、
幾個賣點都行）。依序做 5 步，**每步都讓使用者能改，最後才存檔**：

**Step 1｜整理產品檔**
讀 `templates/product-profile.template.md`，把使用者貼的資訊整理成標準格式：
產品基本資訊、核心賣點、真實可寫進腳本的特色。缺的欄位就標「（待補）」並提醒。

**Step 2｜設定 CPI 目標與轉換目標**
問使用者這支產品的**投放目標**（多為 App 安裝）與 **CPI／CPA 目標門檻**（例如
NT$20）。這是後面判紅綠燈的依據，**不寫死、每支產品自己設**。

**Step 3｜用 CCN 推出 4 個 TA**
依 `ccn-framework.md` 的 TA↔CCN 骨架，把產品翻成 4 個 TA：
- TA-A（Core，最熱）、TA-B（Core/Casual）、TA-C（Casual）、TA-D（New）
- 每個 TA 要給出：①名稱 ②**投放面板真的能設的鎖定條件**（興趣／年齡／受眾）
  ③痛點 ④溝通重點。

**Step 4｜生產品專屬切角庫＋建議第一輪**
把產品賣點對應到 `ccn-framework.md` 的通用切角原型，生出這支產品的候選切角庫，
並**推薦第一輪先測哪個切角＋理由**（通常挑最能打中 Core 痛點、又還沒驗證的）。

**Step 5｜設定平台矩陣＋給投廣策略**
問使用者這支產品**投哪些平台／OS**（例：Meta iOS+Android、Google 只 Android，
或只投 Meta、或含網頁）。據此算出「一輪要追蹤幾隻廣告」。接著給出投廣策略：
- 測試節奏：1 切角 × 4 TA × 1 週
- 平台/OS 矩陣與隻數
- 建議日預算（要能讓每隻在 4–5 天內累積到有意義的 CPI 讀數）
- 停損線（用 Step 2 的 CPI 目標）
- 命名規則

**最後**：把以上彙整，問使用者「要不要建立這支產品的工作區」。**點頭才**：
建 `桌面/切角引擎/<產品名>/`，寫入 `product-profile.md`（Step 1–5 的成果）與
空白 `matrix-state.md`（複製自 template，把 CPI 目標、平台矩陣填進去）。

### 「切換到 XX 產品」／「換產品」
1. 到 `桌面/切角引擎/` 找該產品資料夾，讀它的 product-profile.md 與 matrix-state.md
2. 回報「現在操作的是 XX，CPI 目標 __，已測過哪些切角」，之後動作都對這支
3. 找不到就列出現有產品讓使用者選，或提示可用「新增產品」

### 「發想這一輪的切角」／「發想切角」
1. 確認目前產品 → 讀該產品 `matrix-state.md`，列出已生過的切角（避免重複）
2. 讀該產品 `product-profile.md` 拿產品資訊、4 個 TA、切角庫
3. 讀 `ccn-framework.md` 拿發想邏輯
4. 選一個**還沒測過的新切角**（使用者沒指定就推薦一個並說明理由）
5. 針對 4 個 TA，各生一支 **10 秒短影音腳本**，用 CCN 對應各 TA 的溫度
6. 每支腳本含：切角＋TA、Hook（前 3 秒）、中段、CTA、建議畫面／字卡
7. **先不寫檔**，問使用者要不要調整、要不要存

### 「存進 matrix」／「存進 matrix-state.md」
1. 把這次發想的切角＋4 支腳本寫進該產品 `matrix-state.md` 的「靈感庫」與「腳本存檔」
2. 切角狀態標「⬜ 待測」
3. 輸出可貼上 Google Sheet「靈感庫」分頁的表格

### 成效追蹤表結構（表A 主控台 + 表B 紀錄）— 認表頭不認欄位字母

> ⚠️ **前置設定**：本節起的成效追蹤指令（建立成效追蹤表／採用進表A／補廣告ID／
> 更新 CPI）需要先裝好 **gws CLI ＋ 接上行銷 MCP**。任何一步偵測到 gws 或 MCP
> 不可用時，先引導使用者去看 `references/setup-gws-mcp.md` 完成安裝，再繼續。

這支產品的成效追蹤 = 一張 Google Sheet、兩個分頁。所有讀寫**一律先讀第 1 列表頭、
用「欄位名稱」定位欄**，絕不寫死欄位字母（使用者隨時可能刪/加/搬欄）。**表頭名稱請
勿更改**，改名流程就找不到。Sheet ID 記在該產品 `matrix-state.md`（私人資料、不外流）。

**表A「主控台」= 現況快照，一支廣告一列**（就地更新最新值）
標準欄位：`平台｜TA｜OS｜切角｜廣告命名｜興趣鎖定｜年齡｜日預算｜yt影片連結｜桌面影片名稱｜對應腳本｜廣告ID｜今日CPI｜近7天平均CPI｜建議燈號｜我的決策｜最後更新`
- 自動化**只就地更新**：今日CPI、近7天平均CPI、建議燈號、最後更新。
- 自動化**永遠不碰**：廣告ID、**我的決策**（使用者手填：保留/加碼/關閉）、所有素材欄。

**表B「CPI 紀錄」= 流水帳，只增不改**（每次更新 append 一批）
標準欄位：`更新日｜產品｜切角｜平台｜TA｜OS｜廣告ID｜今日花費｜今日安裝｜今日CPI｜近7天平均CPI｜燈號`

### 「建立成效追蹤表」／「建表」
從零幫這支產品建一張乾淨的追蹤表（新使用者第一次用）：
1. 先確認 gws 可用。不可用 → 引導使用者看 `references/setup-gws-mcp.md` 裝好再來。
2. `gws sheets spreadsheets create` 開新表 → 加第二分頁「CPI 紀錄」→ 兩頁各寫入上面
   **標準表頭**。**只有表頭、不含任何實際資料**。
3. 把新的 Sheet ID 記進該產品 `matrix-state.md`（標明私人、不外流）。
4. 回報 Sheet 連結給使用者。

### 「採用這批素材」／「加進表A」（新一輪上線的銜接）
使用者確認某輪切角的素材後，把這批廣告 append 進表A 主控台：
1. 讀表A 表頭定位欄。
2. 依該產品平台矩陣，為這輪每支廣告 **append 一列**（不覆蓋既有列）：填 平台/TA/OS/
   切角/廣告命名/興趣/年齡/日預算/對應腳本；「建議燈號」先標「待上線」；**廣告ID 留空**。
3. 同步寫進 `matrix-state.md` 測試結果矩陣。
4. 提醒使用者：上架後要「補廣告ID」才能自動追 CPI（見下）。

### 「補廣告ID」／「對應廣告」
把表A 中「廣告ID 空著」的列接上線上廣告：
- **首選**：若廣告由本 skill 用 MCP 建 → 建完當場把 ID 寫回對應列（名稱一致、免比對）。
- **次選（命名比對）**：讀表A 廣告ID 空列的「廣告命名」→ 用 MCP 撈線上清單
  （Meta `list_campaigns`、Google `list_ad_groups`）→ **模糊比對**名稱（去產品前綴、
  容忍小差異）→ **先給使用者一張「表A列 ↔ 找到的廣告＋ID」預覽，確認沒配錯才寫**。
  配錯 ID 會張冠李戴，這道確認必留。寫完就改認 ID、不再靠名字。

### 「更新這一輪的 CPI」／「更新 CPI」（自動版，預設走這個）

> 不用人工查數字：自己抓成效、算 CPI、判燈號，先給預覽，確認後 → 主控台就地更新、
> 紀錄頁 append。遵守原則 6（先預覽、只動該動的、寫完驗證、認表頭不認字母）。

**Step 1｜讀表、定位、收 ID**
確認產品 → 讀該產品 Sheet ID → 讀表A（含表頭）。用表頭定位欄。收集「廣告ID」欄
**有值**的列的 ID（依平台欄或 ID 格式分 Meta campaign／Google ad_group）。廣告ID
空的列跳過，並提醒使用者「那幾支還沒接上線」。

**Step 2｜抓成效（今日 + 近7天）**
- Meta：`get_ad_insights`（level=campaign）分別抓 `today` 與 `last_7d` 的花費與安裝數
  （安裝＝mobile_app_install／omni_app_install；iOS 常因 ATT 嚴重低估，安裝少屬正常）。
- Google：`gads_get_ad_insights`（level=ad_group）抓 `TODAY` 與 `LAST_7_DAYS` 的花費
  （cost_micros÷1e6）與轉換數。
- CPI＝花費 ÷ 安裝數。

**Step 3｜判燈號（依「近7天平均」，較穩）**
- 近7天安裝 < 10 → 🟡 觀察中（CPI 留白）。
- 近7天安裝 ≥ 10 且 近7天平均CPI < 目標 → 🟢 保留。
- 近7天安裝 ≥ 10 且 近7天平均CPI ≥ 目標 → 🔴 超標（只標，不關）。
- 完全抓不到資料（沒投遞）→ ⚠️ 待查，該列不寫值。
（今日CPI 只當「最新參考」，不拿來判燈號。）

**Step 4｜先出預覽表，先不動 Sheet**
列出：平台/TA/OS｜今日CPI｜近7天平均CPI｜舊燈號→新燈號。等使用者說「寫」才動。
ID 對不到、抓不到數據 → 停下來問，不硬寫。

**Step 5｜寫回（主控台就地更新 + 紀錄頁 append + 驗證）**
1. **主控台**：用表頭定位要寫的欄，把該欄「所有列的新值」組成陣列，**整批一次寫**
   （一次 `gws … values update` 涵蓋整欄，或用 `values batchUpdate` 一次涵蓋多欄）。
   ⚠️ **絕不一格一格分開寫**——12 支 × 4 欄若逐格寫是幾十次 API 呼叫，慢又可能撞限。
   只寫「今日CPI／近7天平均CPI／建議燈號／最後更新」，**不碰 廣告ID／我的決策／素材欄**。
2. **紀錄頁**：用 `gws … values append` **append 一批**（每支一列，欄位同表B 標準）。
   **永遠 append，絕不用固定範圍 update，以免蓋掉歷史。**
3. **讀回驗證**：主控台讀回比對；紀錄頁確認新增了正確列數。不符就報錯。

**Step 6｜同步記憶檔**
把最新 CPI 與燈號同步進 `matrix-state.md` 測試結果矩陣
（emoji：🟡觀察中／🟢保留／🔴超標／⚠️待查）。

### 「更新 CPI（手動版，備援）」（＋使用者直接給數字）
（當 Sheet 未建、或 MCP／gws 不通時退回）
1. 把使用者給的 CPI 寫進 `matrix-state.md` 對應列，判燈號。
2. 輸出可貼上 Sheet 的表格，請使用者手動貼。

### 「分析這一輪，建議下一輪測什麼」／「分析這一輪」
1. 讀該產品 `matrix-state.md` 全部數據
2. 分析並回報：哪個 TA 最好（可加預算）、哪個平台/OS 較有效、這切角整體行不行、
   **推薦下一輪測哪個新切角＋理由**
3. 不自動寫檔，等使用者決定下一輪

## 測試方法論

### 一輪 = 1 個切角 × 4 個 TA，為期 1 週
- **只生 4 支腳本**（一個 TA 一支；同一 OS 家族共用同一支素材）
- 上架時複製到各平台/OS，用命名區分。**隻數依產品的平台矩陣**：
  - 例：Meta（iOS+Android）＝4 TA × 2 OS = 8 隻；Google（只 Android）＝4 隻 → 一輪 12 隻
  - 若只投 Meta（iOS+Android）＝8 隻；若某產品只有 Android 單平台則更少
  - 實際隻數在「新增產品」時定好，寫進該產品檔

### 停損規則
```
切角上線 → 跑 4-5 天 → 看每一隻 CPI
  CPI ≥ 該產品目標 → 直接停掉那一隻（🔴）
  CPI <  該產品目標 → 保留繼續觀察（🟢）
```

### 廣告命名規則
```
[產品]-[平台]-[TA]-[切角]-[OS]-[日期]
範例：<產品>-Meta-TA-A-效率-iOS-0705
     <產品>-Google-TA-A-效率-Android-0705
```
> ⚠️ **命名開頭一定要帶產品名**（用該產品 product-profile 的正式名稱或
> 常用簡稱）。內部成效分析平台（metaagent）是靠「campaign 名稱含產品
> 字眼」自動歸類的——沒帶產品名的 campaign 會掉進「未分類」，使用者
> 會找不到。生成「更新命名」的廣告列時也要用這個格式。

## 語氣與風格
- 使用者無技術背景，用白話、具體、舉例。
- 每次動作前先簡短說明要做什麼，再做。
- 主動提醒「該存檔了」「現在在哪支產品」，避免漏記或寫錯產品。
- 腳本要口語、有網感，適合短影音，不要像教科書。
